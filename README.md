# CaseBridge - предсказание стоимости фриланс-задач

ML-модель для прогнозирования цены и ценового диапазона фриланс-задач на платформе CaseBridge на основе трёх входных признаков: типа задачи, технологического стека и срока выполнения. Решение включает EDA, регрессионную модель для предсказания точной цены в рублях и классификационную модель для предсказания ценового диапазона.

## Подход

1. **EDA**: обнаружена сильная скошенность цены вправо, из-за чего к целевой переменной применена log-трансформация. Проверен баланс классов ценового диапазона (класс "5 000–15 000" занимает больше половины датасета), обнаружено, что поле `наименование_задачи` дублирует `тип_задачи` (104 уникальных значения на 3000 строк, каждое соответствует одному типу задачи) и было исключено из признаков.
2. **Подготовка данных**: категориальные признаки закодированы через `pd.get_dummies()`, выборка разделена на train/test 80/20 (`random_state=42`).
3. **Регрессия**: обучены `LinearRegression` (baseline) и `RandomForestRegressor(n_estimators=200)`, метрики считаются в рублях (после обратного преобразования).
4. **Классификация**: редкие верхние классы ("60 000–120 000" и "свыше 120 000") объединены в один, использован `stratify` при разбиении и `class_weight= "balanced"` в `RandomForestClassifier` для борьбы с дисбалансом.

## Результаты

### Регрессия (предсказание цены)

Модель             MAE (руб.)   R^2 

Linear Regression   ~5 831     ~0.79 
Random Forest       ~6 129     ~0.74 

Оба результата превышают минимальный порог R^2 >= 0.55, заданный в ТЗ; линейная регрессия неожиданно показала себя не хуже случайного леса (за счёт log-трансформации, которая сделала практически линейной зависимость цены от признаков)

### Классификация (предсказание диапазона)

`RandomForestClassifier(class_weight="balanced")` обучен на объединённых классах диапазона цены. Итоговая точность (accuracy) на тестовой выборке: около 0.55, при этом наибольшую полноту (recall) модель показывает для доминирующего класса "5 000–15 000", а на редких классах ("60 000+" и "до 5 000") точность заметно ниже: сказывается дисбаланс данных даже после балансировки весов.

### Важность признаков

Наиболее значимым признаком для обеих задач оказался **срок_выполнения_дней**, далее следуют отдельные категории `тип_задачи` (в первую очередь "Мобильное приложение", "Интернет-магазин", "Fullstack-разработка"). Это согласуется с выводами EDA: график scatter-плота срока выполнения показал заметную положительную корреляцию с ценой (коэффициент Спирмена ≈ 0.45), а barh-график медианной цены по типу задачи явно выделил мобильную разработку и интернет магазины как самые дорогие категории.

## Что модель умеет и где ошибается

- Модель хорошо улавливает общий порядок цены и разделяет дешёвые задачи (вёрстка, лендинги) от дорогих (мобильная разработка, fullstack).
- Слабое место: признак `технологический_стек`: 60 уникальных значений при one-hot кодировании превращаются в разреженные колонки, из-за чего конкретный стек внутри одного типа задачи почти не влияет на прогноз.
- В классификации модель хуже определяет редкие ценовые диапазоны - это ожидаемо при сильном дисбалансе классов и относительно небольшом датасете (3000 строк).

## Идеи для улучшения

- **Target encoding для `технологический_стек`** (посчитанный только на train, без утечки) вместо one-hot — должен дать наибольший прирост, так как сожмёт 60 разреженных колонок в один плотный признак со средней ценой по стеку.
- **`cross_val_score(cv=5)`** вместо единственного train/test split - более надёжная оценка модели на небольшом датасете.
- **`GridSearchCV`** по `n_estimators` и `max_depth` для тюнинга `RandomForestRegressor`.
- Дополнительный признак "средняя цена по паре тип_задачи + стек" - потенциально уловит взаимодействие признаков, которое сейчас теряется.
